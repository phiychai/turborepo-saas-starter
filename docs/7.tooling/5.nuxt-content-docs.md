# Nuxt Content Documentation Setup

This guide explains how to configure Nuxt Content to read documentation from the root `docs/` folder.

## Current Setup

The project currently uses a **sync script approach**:

1. **Source**: Documentation files are stored in `/docs` (project root)
2. **Sync**: The `scripts/sync-docs.sh` script copies files to `apps/web/content/1.docs/`
3. **Content**: Nuxt Content reads from `apps/web/content/1.docs/**/*`

### Running the Sync Script

```bash
# Sync docs from root docs/ to Nuxt Content
./scripts/sync-docs.sh
```

This script:
- Copies all markdown files from `docs/` to `apps/web/content/1.docs/`
- Preserves the folder structure
- Updates documentation for web display

## Configuration Files

### Content Collection Definition

The docs collection is defined in `apps/web/content.config.ts`:

```typescript
docs: defineCollection({
  source: '1.docs/**/*',
  type: 'page',
}),
```

### Nuxt Config

The Nuxt Content module is configured in `apps/web/nuxt.config.ts`:

```typescript
content: {
  // Content configuration
},
```

## Alternative: Direct Access (Advanced)

If you want Nuxt Content to read directly from the root `docs/` folder without syncing, you have a few options:

### Option 1: Use Symbolic Link

Create a symbolic link from the content directory to the docs folder:

```bash
# Remove existing 1.docs if it exists
rm -rf apps/web/content/1.docs

# Create symbolic link
ln -s ../../../docs apps/web/content/1.docs
```

**Pros:**
- No sync script needed
- Changes reflect immediately
- Single source of truth

**Cons:**
- Symbolic links can be problematic in some environments
- Git doesn't handle symlinks well in all cases

### Option 2: Configure Multiple Content Sources

Nuxt Content v2 supports multiple content sources. You can configure it in `nuxt.config.ts`:

```typescript
import { resolve } from 'path';

export default defineNuxtConfig({
  content: {
    sources: {
      // Default content directory
      content: {
        driver: 'fs',
        prefix: '',
        base: resolve(__dirname, 'content'),
      },
      // Root docs folder
      docs: {
        driver: 'fs',
        prefix: '/docs',
        base: resolve(__dirname, '../../docs'),
      },
    },
  },
});
```

Then update `content.config.ts`:

```typescript
docs: defineCollection({
  source: 'docs/**/*', // Relative to the docs source
  type: 'page',
}),
```

**Note**: This approach requires Nuxt Content v2+ and may need additional configuration.

### Option 3: Use Relative Paths (Limited Support)

You can try using relative paths in the collection definition:

```typescript
docs: defineCollection({
  source: '../../docs/**/*', // Relative to content directory
  type: 'page',
}),
```

**Note**: This may not work reliably as Nuxt Content paths are typically relative to the content directory.

## Recommended Approach

For this project, the **sync script approach** is recommended because:

1. ✅ **Reliability**: Works consistently across all environments
2. ✅ **Control**: You can control when docs are synced
3. ✅ **Compatibility**: Works with all Nuxt Content versions
4. ✅ **Separation**: Keeps source docs separate from web content
5. ✅ **Build Process**: Can be integrated into build/deploy process

### Automating the Sync

You can add the sync script to your development workflow:

#### Option A: Watch Mode Script

Create a watch script that syncs on file changes:

```bash
#!/bin/bash
# scripts/watch-docs.sh

# Install chokidar-cli if needed: npm install -g chokidar-cli
chokidar 'docs/**/*.md' -c './scripts/sync-docs.sh'
```

#### Option B: Package.json Scripts

Add to `package.json`:

```json
{
  "scripts": {
    "docs:sync": "./scripts/sync-docs.sh",
    "dev": "pnpm docs:sync && nuxt dev",
    "build": "pnpm docs:sync && nuxt build"
  }
}
```

#### Option C: Git Hooks

Add a pre-commit hook to sync docs:

```bash
#!/bin/bash
# .git/hooks/pre-commit

# Sync docs before commit
./scripts/sync-docs.sh
git add apps/web/content/1.docs/
```

## Troubleshooting

### Docs Not Appearing

1. **Run the sync script**:
   ```bash
   ./scripts/sync-docs.sh
   ```

2. **Check file paths**: Ensure files are in the correct location
3. **Check frontmatter**: Docs need proper frontmatter with `title` and `description`
4. **Restart dev server**: After syncing, restart the Nuxt dev server

### Sync Script Issues

1. **Permissions**: Make sure the script is executable:
   ```bash
   chmod +x scripts/sync-docs.sh
   ```

2. **Path issues**: Check that paths are correct for your OS
3. **File conflicts**: Resolve any merge conflicts in `apps/web/content/1.docs/`

### Path Resolution Issues

If using direct access (Option 2 or 3):

1. **Check Nuxt Content version**: Ensure you're using a compatible version
2. **Verify paths**: Use absolute paths with `resolve()` for reliability
3. **Check console**: Look for path resolution errors in the console

## Best Practices

1. **Source of Truth**: Keep `docs/` as the single source of truth
2. **Version Control**: Commit synced files to `apps/web/content/1.docs/` or add to `.gitignore`
3. **Automation**: Automate syncing in your workflow
4. **Documentation**: Document your chosen approach for team members
5. **Testing**: Test documentation changes locally before committing

## Summary

- **Current Setup**: Sync script copies `docs/` → `apps/web/content/1.docs/`
- **Recommended**: Continue using the sync script approach
- **Alternative**: Use symbolic links or multiple content sources (advanced)
- **Automation**: Add sync to build/dev scripts for convenience

For most use cases, the sync script approach provides the best balance of reliability and simplicity.
